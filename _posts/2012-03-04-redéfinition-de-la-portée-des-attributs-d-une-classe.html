En effet, il est possible d'écrire ceci (à vérifier) : {% highlight php %}trait TraitExample { protected $value; public function getValue() { return $this->value; } } class Example { use TraitExample { value as public } } $obj = new Example; $obj->value = 6; var_dump($obj->getValue());{% endhighlight %} Je trouve que c'est quand même casser le principe fondamental de "boite noire" de la programmation orientée objet. Encore qu'on passe d'un attribut/méthode public vers un attribut /méthode privé, pourquoi pas... mais l'inverse me choque. Pas vous ? Ceci dit, je me suis un peu amusé (je ne dis pas que c'est propre) à voir ce qu'on pouvait faire dans cet esprit sous PHP 5.3. C'est possible, grâce à Reflection : {% highlight php %}$attribute = new ReflectionProperty($object, $name); $attribute->setAccessible(true); $attribute->setValue($object, $value); {% endhighlight %} On peut imaginer passer par un objet tampon par exemple : {% highlight php %}Des avis sur la redéfinition de la portée des attributs des traits avec PHP 5.4 ? je suis le seul que ça choque ?
